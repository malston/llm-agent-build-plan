LangGraph
Stateful graph execution

Builds agents as directed graphs where nodes are functions or LLM calls and edges are control-flow conditions. The state object is typed and flows through every node; checkpointers (SQLite, Postgres, Redis) persist state across runs for long-lived or human-in-the-loop workflows. The streaming API emits state deltas at each node, which maps cleanly to your OpenTelemetry instrumentation work.

StateGraph TypedDict state Conditional edges Checkpointers ToolNode Subgraphs Interrupt / HITL LangSmith tracing add_messages reducer Supervisor pattern
Google ADK (Agent Development Kit)
Google-native agent SDK

ADK is Google's opinionated SDK for building agents on top of Gemini models and Vertex AI. It ships with a built-in evaluation framework, multi-agent orchestration patterns (sequential, parallel, loop), tool integration via function calling, and deployment to Vertex AI Agent Engine or Cloud Run. The ADK runner handles the ReAct loop internally — you define agents, tools, and routing logic, not the loop itself.

HuggingFace SmolAgents
Code-first minimal agents

SmolAgents takes a radically minimal approach: the model generates executable Python code as its action output (CodeAgent), rather than JSON tool calls. This reduces token overhead and gives the model full Python expressiveness. A ToolCallingAgent variant provides the conventional JSON tool-call loop. Ships with a ManagedAgent wrapper for multi-agent composition and a GradioUI for quick demos.

How each implements the ReAct loop
LangGraph
Explicit node for "call model", conditional edge to "call tools" or END, ToolNode handles execution, loops back via edge. You own the loop shape.
Google ADK
ADK Runner owns the loop; LlmAgent calls Gemini with tool schemas, routes results through registered tools, feeds observation back to model.
SmolAgents
agent.run() owns the loop; model emits Python, PythonInterpreter executes it, stdout/return value captured as observation.

Deep-build plan

12-week senior track for going from ships agents to rigorously good — back to the tracker →

How to read this

This is not a curriculum. You need reps with feedback on the parts most people skip. Each phase ends in something shippable. If a week feels like coursework, you're off the rails.

The one move

Make everything measurable first, then build hard things under measurement, then go below the API.

Weekly operating rhythm
Deep buildMon–Thu morningsThe current phase's main artifact. Phone in another room.
Research sliceOne full dayPapers, SDK source, others' eval harnesses — timeboxed so it doesn't become tutorial hell.
Write / ship2–3 afternoonsTurn the week's work into a public writeup or repo. Writing is how you find out what you don't understand.
Networking1 afternoonOne outreach, one community engagement. Thin but never zero.
Self-demo + retroFridayWhat shipped, what's now measured, what's next. Out loud, to a peer if you can find one.

Failure modes to avoid
Tutorial hell
If you're following along rather than building your own thing, stop. Timebox the research slice hard.
Skipping evals because they're unglamorous
They're the whole point. Do not let Phase 1 get compressed away.
Polishing one project forever
The Friday demo forces forward motion. Ship rough, measure, move.
Going dark
Keep the thin networking thread alive — three months of silence is expensive to restart.
Pretending the model project is bigger than it is
One narrow fine-tune done honestly beats an ambitious one abandoned.
